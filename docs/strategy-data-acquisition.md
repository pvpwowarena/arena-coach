# Стратегия получения данных из игры — сравнение подходов

Один из ключевых вопросов проекта: **откуда брать сигнал о происходящем в матче?** От ответа зависит ToS-риск, стоимость, сложность реализации и качество подсказок. Ниже — все рассмотренные варианты, сильные/слабые стороны и финальная рекомендация.

---

## A. Аддон + chat-frame mirror (текущий план Phase 3-4)

**Что:** Lua-аддон в клиенте WoW читает события боя через стандартный API (`COMBAT_LOG_EVENT_UNFILTERED`, `ARENA_OPPONENT_UPDATE`, `UNIT_AURA`). Пишет данные в:
- `SavedVariables.lua` — для post-match сводки (на `/reload`/logout).
- скрытый chat-frame с префиксом `[AC|<base64-json>]`, который клиент сам логирует в `Logs/Chat-YYYY-MM-DD.txt` — для realtime (см. ADR-0003).

Bridge на машине игрока tail'ит chat-log + парсит SV, шлёт WSS на backend.

| Параметр | Значение |
|----------|----------|
| Качество сигнала | **Очень высокое** (точные spell_id, длительности, актёры) |
| Лаг realtime | 0.5-2 сек |
| Install у тестера | Аддон + Python-бридж (один раз) |
| ToS-риск | **Зелёная зона.** Это ровно тот же паттерн, что используют Details!, Recount, Warcraft Logs uploader, Weakauras export. Никакого input-injection, никакого чтения памяти. |
| Cost эксплуатации | **Ноль** (только VPS) |
| Сложность реализации | Средняя-высокая (Lua + asyncio bridge + protocol design) |
| Работает на private servers | Да, если приватка не блочит addon-API (Anniversary блочить не будет — это официальный клиент) |

**Особенности:**
- Контроль над тем, какие именно события и в каком формате идут наружу.
- Можно специализировать события под нашу арена-логику (post-trinket detection, sap-stall, и т.д.).
- Игрок может в любой момент выключить аддон → данные перестают идти. Полный контроль на стороне игрока.

---

## B. Прямое чтение WoWCombatLog.txt (без аддона)

**Что:** Клиент WoW имеет встроенную команду `/combatlog`, которая включает запись всего combat-log'а в `Logs/WoWCombatLog.txt` в режиме реального времени. Это стандартная фича клиента, не модификация.

Bridge просто tail'ит этот файл, парсит стандартный формат, шлёт на backend.

| Параметр | Значение |
|----------|----------|
| Качество сигнала | Высокое (но raw-формат, без наших custom-фильтров) |
| Лаг realtime | 0.5-1 сек |
| Install у тестера | **Только Python-бридж**, аддон не нужен |
| ToS-риск | **Зелёная зона.** Используется Warcraft Logs официально. Blizzard сами рекомендуют. |
| Cost эксплуатации | Ноль |
| Сложность реализации | Низкая (документированный формат + tail) |
| Работает на private servers | Да |

**Особенности:**
- Тестеру нужно один раз набрать `/combatlog` или замакросить — типичная boilerplate-команда.
- Не различает арена-специфичные события (нужно отличать арена-матчи от пвп-боёв в open world по контексту).
- Файл может быть огромным (10+ MB за сессию) — нужна аккуратная ротация и обработка.
- **Можно начать без аддона** и добавить аддон позже как «pro»-уровень.

## C. Просмотр Twitch-стрима игрока (без аддона, без bridge'а)

**Что:** Игрок стримит свою игру на Twitch (что многие и так делают). Наш сервис подключается к стриму через Twitch API + видео-инференс (CV/OCR/LLM-vision).

Варианты обработки видео:
- **C1: CV+OCR** — OpenCV для template-matching иконок класса/спеллов в HUD, Tesseract для имён игроков и cast-bar.
- **C2: LLM Vision** — отправлять кадры в Claude Vision / GPT-4V с промптом «опиши состояние арена-матча».
- **C3: Hybrid** — CV для high-frequency признаков (трикеты, КД), LLM для low-frequency сложной интерпретации.

| Параметр | Значение |
|----------|----------|
| Качество сигнала | Среднее (зависит от UI игрока, разрешения стрима, моды на UI) |
| Лаг realtime | **3-7 сек** (Twitch ingest buffering + inference) |
| Install у тестера | **Ноль** — только OBS+Twitch (что у стримеров уже есть) |
| ToS-риск | **Зелёная зона на нашей стороне** — мы не запускаем код в клиенте. Twitch ToS требует согласия стримера. |
| Cost эксплуатации | **$1-10/час стрима для CV+OCR**; **$10-50/час для LLM Vision**; **$0.5-5/час hybrid** |
| Сложность реализации | Высокая (CV-пайплайн + калибровка под разные UI + handling fps drop'ов и качества стрима) |
| Работает на private servers | Да (мы видим только видео) |

**Особенности:**
- 5-секундный лаг неприемлем для in-match подсказок («нажми kidney сейчас»), но OK для post-trinket («через 8 сек ожидай готовности trinket'а»).
- UI игрока — главная проблема: одни играют со стандартным UI, другие — Bartender/ElvUI/etc. CV-модель надо калибровать на каждого.
- Гибридный подход (CV для трикетов + LLM для общей картины) — реалистичная середина по затратам.

## D. MCP (Model Context Protocol)

**Что:** В контексте твоего вопроса — это, видимо, либо (а) Claude/OpenAI компьютер-use агент, который смотрит на экран и принимает решения, либо (б) MCP-сервер, через который LLM получает данные.

В нашем случае MCP — это **транспорт**, а не источник. То есть это инструмент, с которым уже подключены источники данных A/B/C/D. Не «откуда брать данные», а «как их подать LLM'у».

| Параметр | Значение |
|----------|----------|
| Совместимость с нашим стеком | Высокая (Anthropic SDK поддерживает MCP) |
| Замена для A/B/C | Нет, MCP — это слой поверх любого из них |
| ToS-риск | Зависит от источника, не от транспорта |
| Cost | Дополнительный inference-cost от LLM |
| Сложность | Низкая (MCP-обвязка прямолинейна) |
| Когда добавить | Phase 4+ как улучшение LLM-оркестратора |

**Это не отдельный путь — это слой архитектуры.**

## E. Гибрид: Combat-log → upgrade на аддон → Twitch как fallback

**Что:** Поэтапная стратегия:
1. **Phase 4a (быстрый старт):** Bridge читает `WoWCombatLog.txt` через `/combatlog`. Нулевой install у тестеров (только Python-бридж). Базовые подсказки работают.
2. **Phase 4b (улучшенный):** Игроки, которым нужен realtime ≤1 сек и custom-события (post-trinket detection, sap-stall hint), ставят наш аддон.
3. **Phase 5+ (Twitch fallback):** Для игроков, которые не хотят ставить вообще ничего на свой PC, или для просмотра матчей других игроков (которых можно «коучить» без их участия) — стримовая обработка через hybrid CV+LLM.

| Параметр | Значение |
|----------|----------|
| Качество сигнала | Высокое (комбинация) |
| Лаг | 0.5-1 сек для combat-log/аддона; 3-7 сек для стрима |
| Install | Опционально (от «ничего» до «аддон+бридж») |
| ToS-риск | Зелёная зона на всех слоях |
| Cost | Базовый + опциональный CV-cost при использовании стримов |
| Сложность | Средняя (3 источника, но независимы — можно вкатывать по очереди) |

## Рекомендация

**Идём по варианту E (гибрид с поэтапным upgrade'ом):**

1. **Phase 4a:** Bridge читает `WoWCombatLog.txt` (вариант B). Тестеру нужен только Python и наш bridge. Это **MVP с минимальным барьером входа**.

2. **Phase 4b:** Аддон (вариант A) для игроков, которым нужны custom-события и латенси ≤1 сек. Опционально.

3. **Phase 5+:** Twitch-обработка (вариант C) — для observer-режима (коучинг чужих игр без их участия) и для тестеров без возможности поставить bridge.

**Что выкидываем из плана:** ничего. Просто меняем порядок:
- ❌ Phase 3 (Lua-аддон как первый) → ✅ Phase 4a (combat-log как первый, аддон позже)
- ❌ Phase 4.5 voice (Discord voice + TTS) → переносим в Phase 4c (после combat-log + базовых hint'ов)
- ❌ Phase 5 CV/OCR → ✅ Phase 5 Twitch (если будет нужно)

**Почему так лучше:**
- Снижает барьер входа для тестеров на **80%** (только bridge вместо bridge+аддон).
- Дает первый рабочий end-to-end быстрее (3 недели vs 6).
- Аддон можно добавить позже, не ломая bridge.
- Twitch как «оп-режим» — отдельный продукт для commentary-стримеров.

## Анти-бан — что мы НЕ делаем

Все четыре варианта (A/B/C/E) сводятся к **read-only** наблюдению. Что **запрещено** Blizzard ToS и что мы **никогда** не делаем:

| Нарушение | Что это | Где у нас |
|-----------|---------|-----------|
| Input injection | Нажимать клавиши/мышь за игрока | ❌ Никогда |
| Memory reading | Читать память процесса WoW.exe | ❌ Никогда |
| Packet inspection | Перехватывать сетевые пакеты клиента | ❌ Никогда |
| Bot automation | Авто-кастовать спеллы / двигаться | ❌ Никогда |
| File modification | Менять файлы клиента | ❌ Никогда |

Что мы **делаем** — это в точности то, что делают индустриальные инструменты:
- **Аддон** использует только secure-API (тот же API, что Bartender, Details!, Weakauras, BigWigs).
- **Combat-log** — стандартная фича клиента (`/combatlog`), используется Warcraft Logs.
- **Bridge** читает файлы клиента (как Warcraft Logs Uploader, как Method app).
- **Stream-watch** — third-party видео-анализ, никакого взаимодействия с клиентом.

**Прецеденты:** Warcraft Logs, Method, Details!, Weakauras, Bartender, ElvUI — все живы много лет. Никто из них не получал бан-волн от Blizzard за используемые подходы.

## Что я хочу обсудить с тобой

1. **OK с переключением порядка Phase 3 ↔ Phase 4a?** То есть начинаем с combat-log + bridge, аддон откладываем на Phase 4b после первых hint'ов работают.
2. **Twitch-режим (вариант C) — приоритет?** Если у тебя тестеры уже стримят на Twitch, может быть смысл сделать его быстрее. Если нет — отложим до Phase 5.
3. **MCP как слой** — добавляем когда LLM-оркестратор стабилизируется (Phase 4).
