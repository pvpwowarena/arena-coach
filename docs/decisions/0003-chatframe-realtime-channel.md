# ADR-0003: Chat-frame mirror как realtime-канал addon → bridge

- **Status:** Accepted
- **Date:** 2026-05-12
- **Deciders:** Vladislav, Arena Coach Dev
- **Tags:** addon, bridge, realtime, tos

## Context

Lua API в WoW BC Classic 2.4.3 — это secure sandbox. Возможности взаимодействия аддона с «внешним миром» строго ограничены:

1. **`SavedVariables.lua`** записывается на диск только при `/reload` и logout. Это значит, что без действия игрока bridge не получит данные в течение матча — а нам нужны in-match подсказки в Phase 4.

2. **Прямых сетевых функций нет.** Никаких `socket`, `HTTP`, ничего. Это закрыто sandbox'ом и ToS.

3. **Single-channel exception:** аддон может писать в *chat-frame* (стандартное UI-окно сообщений). Клиент WoW при включённой опции `/console chatLogging 1` пишет содержимое всех чат-фреймов в файл `World of Warcraft/Logs/Chat-<date>.txt` в real-time (с лагом ~0.5-2 сек, ограниченным дисковой buffered IO).

Этот файл — стандартный артефакт клиента, не модификация и не патч.

## Decision

**Канал realtime-телеметрии addon → bridge:**

1. Аддон создаёт скрытый chat-frame (`AC_EventChannel`) и пишет в него каждое значимое событие в виде строки:
   ```
   [AC|<base64-encoded compact-JSON>]
   ```
   Префикс `[AC|` — для bridge-фильтра, base64 — чтобы избежать проблем с эскейпингом, JSON — для структуры.

2. Игрок включает `/console chatLogging 1` (однократно, сохраняется в WTF). Это документированная пользовательская опция игры.

3. Bridge на той же машине делает `tail -f`-style чтение файла `Logs/Chat-YYYY-MM-DD.txt` (`watchdog` + line-buffered read).

4. Bridge парсит строки с префиксом `[AC|`, декодирует, валидирует через pydantic, отправляет по WSS на backend с bearer-токеном.

5. SavedVariables остаётся вторым каналом для **детального post-match summary** (тело матча целиком: список всех каст-эвентов, длинные payload'ы, которые не уместятся в строке chat-log'а). Эти данные синхронизируются на `/reload` или logout, и бэкенд использует их для пост-матчевого разбора и обновления статистики, не для real-time подсказок.

## Alternatives Considered

### A. Только post-match через SavedVariables
- **Плюсы:** ничего лишнего, addon не пишет в чат вообще, нулевой риск ToS.
- **Минусы:** нет realtime-подсказок. Phase 4 теряет смысл — это половина продукта.
- **Вердикт:** отвергнуто. Сохраняется как fallback-режим, активируемый флагом `ARENA_COACH_DISABLE_CHAT_MIRROR=1`.

### B. External memory reading
- **Плюсы:** реальный realtime, любые поля.
- **Минусы:** **прямо нарушает Blizzard ToS.** Чтение памяти процесса — основание для бана. Не обсуждается.
- **Вердикт:** **отвергнуто навсегда.**

### C. UDP/named-pipe из addon через локальный плагин
- **Плюсы:** низкий лаг.
- **Минусы:** нативные сетевые/IPC API недоступны Lua-аддону в TBC 2.4.3.
- **Вердикт:** технически невозможно без модификации клиента.

### D. Screen capture + OCR (Phase 5)
- **Плюсы:** не зависит от addon-API вообще.
- **Минусы:** ресурсоёмкость; fragile к UI-изменениям; зона ToS-неопределённости — отдельный risk assessment перед стартом.
- **Вердикт:** отложено до Phase 5, не заменяет chat-frame канал.

## Consequences

**Позитивные:**

- Realtime-подсказки становятся технически возможны.
- Канал read-only: addon только пишет в свой UI; клиент только пишет в свой Log-файл. Никакой модификации игрового процесса.
- Симметрично с тем, как работают другие крупные аддоны (BadBoy, Details!, Recount — все используют chat-frame mirror или SV в той или иной мере).

**Негативные / trade-off'ы:**

- Игрок обязан включить `/console chatLogging 1` — это explicit-step в setup'е.
- Лаг 0.5-2 сек. Приемлемо для подсказок «друид трикетнул → готовь повторный stun», неприемлемо для подсказок «нажми кнопку сейчас» (которые мы и не делаем).
- Размер одной строки chat-log'а ограничен (~255 байт визуально, ~~16 KB технически в TBC). Это значит большие event'ы (например, длинный список aura'ов с payload'ом) идут через SV, не через chat-mirror.
- Файл chat-log'а ротируется по датам; bridge обязан корректно подхватывать новый файл при переходе через полночь.

## ToS оценка

- Аддон не модифицирует игровой стейт, не нажимает кнопки, не делает action на игрока (`secure templates` гарантируют это на уровне API).
- Аддон не читает память процесса, не использует CVar'ы вне документированных.
- Bridge — внешний процесс, не взаимодействующий с клиентом WoW напрямую: читает только текстовый log-файл, генерируемый стандартной опцией игры.
- Тождество с тем, что делают legal-комбат-логгеры (Warcraft Logs uploader, Details! exporter): они тоже tail'ят файлы клиента.

**Перед Phase 4** — отдельный документ `docs/tos-risk-assessment.md` с финальной проверкой, including краткой записью «что НЕ делаем» (auto-action, memory inject, screen-overlay-input-capture).

## Validation

Phase 3 (после написания addon'а):
1. Включить `chatLogging 1`, запустить addon, провести тренировочный матч.
2. Открыть `Logs/Chat-*.txt`, убедиться что строки `[AC|...]` появляются с лагом ≤2 сек после события.
3. Bridge-stub читает эти строки и логирует их в stdout.

Если на этом шаге выясним, что клиент Anniversary иначе обрабатывает chat-log (например, лаг 10 сек или не пишет вообще) — пересматриваем ADR.

## Rollback plan

Если Phase 3 покажет, что chat-mirror не работает (или сообщество получит явный сигнал от Blizzard, что подобные каналы запрещены) — переходим на режим B (`/reload`-trigger через post-match summary), Phase 4 переориентируется на post-match coaching, in-match подсказки удаляются из scope v1.
